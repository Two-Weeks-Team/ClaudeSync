2. 핵심 발견 — "페어링이 작동 안 하는 이유"

사용자가 "지난 세션에서 에러를 본 적 있는데 기억이 흐리다"고 진술. 4명의 전문가 중 root-cause-analyst가 **3개의 ship-blocking issue**를 동시에 발견했고, 이것이 사용자가 봤을 증상의 정체로 추정됩니다.

🚨 RCA-C1: Onboarding 페어링 버튼 = silent no-op
FirstLaunchPairingView가 자기 안에서 OnboardingViewModel()을 새로 만들어 사용했고, 그 모델의 onAcceptPair / onConfirmPair / onRejectPair 클로저는 코드베이스 어디에서도 set되지 않음. 사용자가 "Accept", "Confirm", "Cancel"을 눌러도 아무 일도 안 일어남.
🚨 RCA-C2: Responder Mac에 incoming pairing UI가 아예 없음
Mac A가 Mac B에게 "Pair" 클릭하면 B의 acceptIncomingPairing가 PairingManager를 시작하고 받아들이는 사용자의 confirm을 영원히 대기. 그런데 B의 어디에도 그 confirm UI가 없음 — 메뉴바 popover는 outgoing Pair 버튼만 있고, onboarding 윈도우는 RCA-C1 때문에 어차피 미동작.
🚨 RCA-C3: 페어링 결과가 영속화되지 않음
페어링 성공해도 activePairedPeer가 메모리 var에만 저장됨. 앱 재시작 = 매번 재페어링 필요. authorized_keys는 디스크에 남아있지만 앱이 그 사실을 모르고 FileSyncActor.peer = nil로 부팅 → 모든 sync job이 "no peer configured" 실패.

v1.0.1에서 정상화된 흐름

  • AppEnvironment가 OnboardingViewModel을 소유하고 init 시점에 콜백 wiring → 버튼이 실제 PairingManager 호출
  • MenuBarRootView에 pairingBanner 섹션 추가 → 모든 pairing 단계 (request 수신, accept 수신, sentAccept 대기, failed, rejected)에서 코드 + Accept/Cancel 버튼 노출
  • Onboarding discovery step에 발견된 peer 인라인 표시 + Pair 버튼 (사용자가 메뉴바 닫고 다시 열 필요 없음)
  • Preferences.pairedPeer 영속 필드 추가, handlePairingCompleted에서 저장, bootSyncEngine에서 복원 → FileSyncActor 즉시 재배선
  • Settings → Forget paired peer 버튼 추가

4.2 Sync loop 차단 (CR-C2 / CR-C3 / RCA-M11)

⚠️ 사용자가 봤을 가능성 가장 높은 증상
v1.0-rc1의 PID 기반 echo suppression은 수신측 rsync가 sshd의 child라 PID 등록 불가 → 두 Mac이 모든 파일 변경마다 무한 ping-pong. v1.0.1은 mtime 기반으로 전환:
ID패치 파일설명
CR-C2 FileWatcherActor.swift mtime-stale echo filter: rsync --archive가 source mtime을 보존하므로, FSEvents가 5초 이상 오래된 mtime의 파일에 대해 발사되면 echo로 판단해 drop. user 편집은 항상 mtime ≈ now.
CR-C3 FileSyncActor.swift rsync exit 0 + stdout itemize-changes 파싱 → 진짜 transferred 카운트로 false-success 차단
RCA-M11 ProcessRunner.swift real PID 노출 + UUID 기반 마커 PID로 collision 방지
