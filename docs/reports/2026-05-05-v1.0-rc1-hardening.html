4. 핵심 결정 — "ps RSS 91MB 트랩"과 10인 전문가 패널

측정 도구의 정확도가 PRD 목표 검증의 신뢰성을 결정한 사건입니다.

4.1 측정 시퀀스

  1. 최초 측정: ps -o rss = 91.6 MB → "PRD G4 50MB 초과!"
  2. 10인 전문가 패널 스폰: Christensen, Porter, Drucker, Godin, Kim/Mauborgne, Collins, Taleb (비판적), Meadows, Doumont, macOS Sys Engineer
  3. 다수결: A=0(차단) · B=5(PRD 갱신) · C=4(검증 후 결정) · C+B=1 → C → B 하이브리드
  4. 30분 timebox 검증: heap PID 명령 실행
  5. 실 결과: Physical footprint = 22.8 MB, peak 22.9 MB, 0 leaks

4.2 진단

ps -o rss는 macOS에서 SwiftUI · AppKit · Network.framework · CoreServices 등 모든 공유 라이브러리 페이지를 합산해 보고합니다. Apple이 메모리 압력 지표로 사용하는 것은 Activity Monitor "Memory" 컬럼과 동일한 Physical footprint이며, 이 값이 PRD G4의 의도와 일치합니다.

⚠️ Taleb 우려 해소
Taleb는 "PRD를 측정값에 굴복시키는 것은 Goodhart's law의 역방향이며 fragility를 코드에 잠그는 행위"라고 비판했습니다. 측정 도구의 정확도부터 검증한 결과, PRD를 갱신할 필요 없이 실측 22.7MB << 50MB 목표로 충족됨이 확인되었습니다. 측정값이 의심될 때는 측정 도구부터 의심하라는 원칙이 시스템에 학습되었습니다.
✅ 결정 산출물
scripts/measure-footprint.sh는 heap 기반 Physical Footprint를 1차 지표로, ps RSS는 비교용 보조 지표로 같이 기록하도록 갱신되었습니다. soft fail 임계는 50MB, hard fail은 75MB.

7. 알려진 제약 / 후속과제

  • AppIcon은 placeholder — Core Graphics gradient + sync glyph로 자동 생성. v1.1에서 디자이너 작업물로 교체 권장.
  • LaunchAtLoginTests는 실 SMAppService.register 회피 — LaunchServices에 시스템 부작용. 실 등록/해제 검증은 사용자 수동.
  • Settings Excludes 탭 동작 — 변경 즉시 builder를 swap하지만, 이미 큐에 들어간 SyncJob은 이전 builder로 실행됨 (의도적 — 진행 중인 sync 보호). 다음 큐부터 새 패턴 적용.
  • DMG는 ad-hoc 서명만 — Gatekeeper가 다운로드-앤-오픈 차단. Developer ID 서명+공증 후 배포 가능.
  • Persistent agent (b5zrojfgw 등) 정리 — 이전 세션의 stale background bash 작업이 남아 있을 수 있음. 신규 세션 시작 시 jobs로 점검 권장.
